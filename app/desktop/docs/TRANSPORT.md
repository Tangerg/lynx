# Lyra Transport Layer

> How the wire protocol (defined in `API.md`) physically travels between a
> Lyra frontend and a Lyra backend. The protocol stays the same across all
> deployment shapes; the **transport** is whatever moves bytes (or, in the
> Go-to-Go case, struct pointers) between the two halves.
>
> Audience: anyone shipping a new frontend variant, embedding the backend
> into a new shell, or writing a thin adapter (HTTP → in-process, socket →
> in-process, etc.) for a specific deployment.

---

## 1. The m × n matrix

Three backend deployment modes × three frontend variants = nine viable
combinations. Each cell picks the **cheapest transport that satisfies the
process / network boundary** of that combination.

| ↓ Frontend / Backend → | Local (loopback) | Server (LAN / WAN) | Embedded (same process) |
| --- | --- | --- | --- |
| **TUI (Go, e.g. Bubble Tea)** | Socket / HTTP | HTTP | **In-process** |
| **Packaged client (Wails / Tauri / Electron)** | Socket / HTTP | HTTP | Wails IPC / HTTP |
| **Web (browser)** | HTTP | HTTP | n/a (no embedding) |

Reading the matrix:

- **Local**: backend runs as a separate OS process on the same machine
  (e.g. `lyra-server` as a systemd unit, or a per-user daemon). Frontend
  reaches it via Unix socket / Named pipe (cheapest) or loopback HTTP.
- **Server**: backend is a remote service. Only HTTP works — everything
  else assumes co-residency.
- **Embedded**: backend code is linked into the same binary as the
  frontend. If both are Go, the boundary is a function call; if the
  frontend is a WebView (Wails / Tauri / Electron) the boundary is the
  bridge IPC the shell already provides.

All six unique transports share **one Go interface**; the handler code
(`Router.Dispatch(req)`) doesn't change between them.

## 2. One abstraction — `Transport`

Both halves talk to a single Go interface. The frontend gets a `Client`
that wraps a `Transport`; the backend gets a `Server` that wires a
`Transport` to a `Router`. **Everything else — HTTP framing, SSE parsing,
socket buffers — is hidden behind this interface.**

```go
// pkg/transport/transport.go
package transport

import "context"

// Request is the wire payload — a method name plus opaque bytes.
// For Go-to-Go in-process the bytes are skipped (see InProcessTransport).
type Request struct {
    Method  string            // "v1.session.send", "v1.message.list", ...
    Headers map[string]string // auth, idempotency-key, trace context
    Body    []byte            // JSON-encoded per API.md, OR nil for in-process
    Stream  bool              // true → response is an event stream (SSE-equivalent)
}

type Response struct {
    Status  int               // mirrors HTTP semantics even when not on HTTP
    Headers map[string]string
    Body    []byte            // for non-streaming
    Events  <-chan Event      // populated when Request.Stream == true
}

type Event struct {
    ID   string  // monotonic per stream — replay key
    Type string  // "TEXT_MESSAGE_CONTENT", "RUN_FINISHED", ...
    Data []byte  // JSON-encoded event payload
}

// Transport is what every adapter implements. One interface, six
// implementations (in-process, Wails, Unix socket, named pipe, HTTP, WS).
type Transport interface {
    // Call performs one request. Honors ctx cancellation. For streaming
    // requests, Response.Events is non-nil and the caller drains it.
    Call(ctx context.Context, req *Request) (*Response, error)
    // Close releases any underlying resources (sockets, channels, ...).
    Close() error
}
```

The frontend never sees `*http.Request` or `net.Conn`. The backend's
handler signature is `func(ctx, *Request) (*Response, error)` — same code
runs behind every transport.

## 3. The service contract — `pkg/coreapi`

`Transport` carries opaque bytes. The **typed Go interface** that both
ends agree on lives one layer up, in `pkg/coreapi`:

```go
// pkg/coreapi/coreapi.go
package coreapi

import "context"

type CoreAPI interface {
    // Sessions
    ListSessions(ctx context.Context, q ListSessionsQuery) (*Page[Session], error)
    CreateSession(ctx context.Context, in CreateSessionIn) (*Session, error)
    // Messages
    ListMessages(ctx context.Context, sessionID string, q ListMessagesQuery) (*Page[Message], error)
    // Runs (the streaming surface)
    StartRun(ctx context.Context, in StartRunIn) (<-chan RunEvent, error)
    CancelRun(ctx context.Context, runID string) error
    // HITL
    SubmitApproval(ctx context.Context, in ApprovalIn) error
    // ... full surface mirrors API.md §4
}
```

This interface is the **single source of truth for behavior**. Two
artifacts derive from it:

- **OpenAPI 3.1 + AsyncAPI 2.6** files (`schemas/`) — generated by
  scanning struct tags, used by non-Go clients (TS, Rust, Python).
- **HTTP server adapter** (`pkg/httpserver`) — routes incoming `*Request`
  to `CoreAPI` methods, marshals the result back into bytes.

So:

```
              ┌───────────────────┐
   Frontend ──│   CoreAPI client  │── uses Transport ──→ wire
              └───────────────────┘
                                                            ↓
              ┌───────────────────┐
   Backend  ──│  CoreAPI impl     │── exposed by Adapter ─→ wire
              └───────────────────┘
```

The adapter shape varies per transport:

| Transport | Adapter shape |
| --- | --- |
| InProcess | The frontend's `Client` holds a direct `CoreAPI` reference. No adapter — `client.ListSessions(...)` is `impl.ListSessions(...)`. |
| Wails / Socket / HTTP | Adapter marshals `*Request` ↔ `CoreAPI` method calls. |

## 4. The six transport implementations

### 4.1 InProcessTransport — Go ↔ Go, same binary

**When**: Bubble Tea TUI linked against `pkg/coreimpl/`, or a Wails app
where we choose to bypass the WebView bridge and write a Go-only path.

```go
// pkg/transport/inprocess/inprocess.go
type InProcessTransport struct {
    api coreapi.CoreAPI // direct reference, no serialization
}

func (t *InProcessTransport) Call(ctx context.Context, req *Request) (*Response, error) {
    // For in-process we DON'T marshal through Request.Body — the client
    // wrapper bypasses bytes entirely and calls api.* directly. This
    // implementation exists only for codepaths that genuinely need the
    // generic Request shape (e.g. shared logging middleware).
    return dispatchInProc(ctx, t.api, req)
}
```

In practice the **Go client wraps `CoreAPI` directly** for the in-process
case — no `Request`/`Response` round-trip needed:

```go
// pkg/coreclient/inproc.go
func NewInProcClient(api coreapi.CoreAPI) coreapi.CoreAPI {
    return api // same interface, returned as-is
}
```

That's the entire savings: **no JSON, no bindings, no codegen**. Bubble
Tea's `tea.Cmd` returns `tea.Msg`; we make `RunEvent` implement `tea.Msg`
and stream straight from `StartRun` into the TUI's update loop.

Cost: ~50 ns / call (Go interface dispatch). Streaming throughput limited
only by channel buffering.

### 4.2 WailsTransport — packaged-web shell ↔ embedded Go backend

**When**: Wails / Tauri / Electron with the backend in the host process,
frontend in the WebView.

```go
// pkg/transport/wails/wails.go
type WailsTransport struct {
    runtime wails.Runtime
}

func (t *WailsTransport) Call(ctx context.Context, req *Request) (*Response, error) {
    // Wails IPC: WebView posts the Request as a message, host routes it
    // to a registered handler. For Stream=true we use Wails' EventsEmit
    // to push events back to the WebView, where a thin shim reassembles
    // them into a channel-like iterable for the TS client.
    ...
}
```

Frontend side is a TS shim that exposes the same `Transport` shape:

```ts
// frontend/src/transport/wails.ts
export const wailsTransport: Transport = {
  async call(req) {
    if (req.stream) {
      const events = makeEventStream(req.method); // wraps Wails EventsOn
      window.go.transport.Stream(req); // fire-and-forget; events flow back
      return { status: 200, headers: {}, events };
    }
    return window.go.transport.Call(req);
  },
  close() {},
};
```

Cost: ~30 µs / call (Wails IPC + structured-clone). Avoids HTTP framing
+ TCP stack entirely while keeping the WebView untouched.

### 4.3 SocketTransport — cross-process loopback (Unix socket / Named pipe)

**When**: TUI in language X talks to a Go backend running as a sibling
process. Cheaper than HTTP loopback (no TCP, no headers parsing), but
still crosses a process boundary so requires marshalling.

```go
// pkg/transport/socket/socket.go
type SocketTransport struct {
    addr string // "/tmp/lyra.sock" on Unix, `\\.\pipe\lyra` on Windows
    conn net.Conn
}

func (t *SocketTransport) Call(ctx context.Context, req *Request) (*Response, error) {
    // Wire format: length-prefixed JSON. For streaming, each event is a
    // separate length-prefixed frame on the same conn until end-of-stream.
    ...
}
```

Wire format choice: **length-prefixed JSON** (varint length + body) over
the socket. Simple to implement in any language, no framing ambiguity,
streams are just N frames in a row. Same serialization as HTTP body — TS
/ Rust / Python clients reuse the schema-generated types.

Cost: ~30 µs / call. Cross-platform: `unix:///tmp/lyra.sock` on macOS /
Linux, `npipe:///./pipe/lyra` on Windows. Auth via socket file mode
(Unix) or named-pipe ACL (Windows) — no Bearer token needed for
loopback-only.

### 4.4 FetchTransport — anywhere → anywhere, HTTP

**When**: web browser, TUI talking to remote server, packaged-web client
talking to managed cloud, or any cross-machine combination.

```ts
// frontend/src/transport/fetch.ts
export const fetchTransport = (baseURL: string, token: string): Transport => ({
  async call(req) {
    const res = await fetch(`${baseURL}${methodToPath(req.method)}`, {
      method: methodToVerb(req.method),
      headers: { Authorization: `Bearer ${token}`, ...req.headers },
      body: req.body,
    });
    if (req.stream) {
      return { status: res.status, headers: hdrs(res), events: parseSSE(res.body!) };
    }
    return { status: res.status, headers: hdrs(res), body: new Uint8Array(await res.arrayBuffer()) };
  },
  close() {},
});
```

Backend side: `pkg/httpserver/` is a thin adapter mapping HTTP verbs +
paths → `CoreAPI` methods, then re-encoding the response.

Cost: ~200 µs / call locally, ~20–200 ms WAN. Required for browser
(can't use sockets), required for remote (only thing that traverses
firewalls / proxies cleanly).

### 4.5 WebSocketTransport — fallback for browsers that need bidirectional

**When**: web frontend that needs to push events upstream mid-run
(uncommon — HITL approval can go via plain `POST /permission` and SSE
covers downstream).

Listed for completeness; **not implemented in MVP**. SSE + REST covers
everything except true client-push-during-server-streaming, which
nothing in the current protocol needs.

### 4.6 Choosing per combination

| Combination | Recommended transport | Why |
| --- | --- | --- |
| Bubble Tea TUI + embedded Go runtime | **InProcess** | Same binary, both Go — direct function call beats anything else by 600×. |
| Wails client + embedded Go runtime | **Wails IPC** | Already paid for by Wails; no extra port; no firewall popup on macOS. |
| Wails client + local server | **Socket** | Cross-process but co-machine; socket is 6× faster than loopback HTTP and skips Bearer-token plumbing. |
| Wails client + remote server | **HTTP** | Only thing that traverses network. |
| Browser + any server | **HTTP** | Only option the browser exposes. |
| Node / Rust / Python TUI + embedded | n/a | These languages can't embed our Go runtime in-process. Use Socket against a co-spawned `lyra-server` subprocess. |
| Node / Rust / Python TUI + local | **Socket** | Same reasoning as Wails-local. |
| Any client + remote / managed | **HTTP** | Network. |

Rule of thumb: **never serialize when you don't have to**. In-process is
free, IPC is cheap, sockets are decent, HTTP is the ceiling.

## 5. Streaming — uniform semantics across all transports

Every transport exposes the same primitive: `Response.Events`, a Go
channel (or its TS equivalent — async iterator) yielding `Event` values
with monotonic `id`, `type`, `data`. This means **resume / replay is
implemented once, in the protocol layer, not per-transport**.

| Transport | Underlying mechanism |
| --- | --- |
| InProcess | Go channel of `RunEvent` |
| Wails | `EventsEmit` + JS reassembly into `AsyncIterator<Event>` |
| Socket | Length-prefixed frames on the conn until end-of-stream sentinel |
| HTTP | SSE (`text/event-stream`) with `id:`, `event:`, `data:` |

The client's resume logic — "reconnect, send `Last-Event-ID: <last>`,
backend replays from event N+1" — works identically against every
transport because they all surface the same `Event.ID` ladder.

For in-process, "resume" is degenerate (the channel is process-local and
dies with the process), so the client simply checks if its `<-chan
Event` is closed and restarts. Same code path, different physics.

## 6. Auth & boundaries

| Transport | Auth model |
| --- | --- |
| InProcess | None — same process trust. Caller capability checks happen inside `CoreAPI` impl, not at transport boundary. |
| Wails | None on the wire (Wails IPC is window-scoped); `CoreAPI` impl still enforces capability checks. |
| Socket | File mode (Unix) / ACL (Windows). Connecting user must own the socket file. No Bearer token. |
| HTTP local | **Bearer token required.** Token generated at embedded-backend install time, written to `~/.config/lyra/token` (chmod 600). Without this, any process on the box could call us. |
| HTTP remote | OAuth (managed cloud) or static API key (self-hosted). |

The `Request.Headers["authorization"]` slot exists for HTTP-class
transports; in-process and Wails leave it empty. `CoreAPI` impls **never
trust the transport** — capability checks (does this user own this
session?) happen inside the impl, regardless of how the call arrived.

## 7. Schema flow — one contract, multiple emitters

```
  pkg/coreapi/*.go  (Go interface + structs)
            │
            ├── go-openapi codegen ──→  schemas/openapi.yaml
            │                                  │
            │                                  ├── openapi-typescript ──→ frontend/src/lib/api-types.ts
            │                                  └── oapi-codegen      ──→ pkg/httpserver/generated.go
            │
            └── go-asyncapi codegen ──→  schemas/asyncapi.yaml
                                               │
                                               └── asyncapi-codegen-ts ──→ frontend/src/lib/events.ts
```

For Go ↔ Go (in-process / Wails), **no codegen is needed** — both sides
import `pkg/coreapi` directly. For everything else, the schemas are the
contract and the TS / Rust / Python clients are generated.

## 9. Why this carves out cleanly

- **Adding a transport doesn't touch handler code.** New variant — say,
  gRPC for inter-service calls in the managed cloud — implements
  `Transport`, registers a new adapter that calls `CoreAPI`, done.
- **Adding a frontend doesn't fork the backend.** New variant — say, a
  Tauri rewrite of the desktop client — picks the transport that fits
  its shell (Tauri IPC ≈ Wails IPC; falls back to HTTP if no embedded
  backend), implements the TS-side `Transport`, reuses the rest.
- **The Go-to-Go case isn't a special snowflake.** It uses the same
  `Transport` shape, but the implementation is empty wrapping: the
  client just holds the `CoreAPI` value. Bubble Tea, internal CLI
  tools, future Go-native plugins all use the same path.
- **HTTP isn't privileged.** `pkg/httpserver/` is one adapter among
  four. The protocol doesn't bake HTTP-isms into `CoreAPI` (no `Request
  *http.Request` parameters, no `http.ResponseWriter` returns). Test
  with in-process; production picks whatever the deployment needs.

## 10. Open questions

- **gRPC?** Would fit `Transport` cleanly but adds proto schema as a
  second SSOT next to OpenAPI/AsyncAPI. Defer until a non-MVP customer
  asks.
- **WebSocket vs SSE for HTTP streaming?** SSE today. WS only if HITL
  needs mid-run client → server pushes (currently it doesn't —
  `/permission` is a plain POST).
- **TUI in non-Go languages — embed or spawn?** No way to embed our Go
  runtime in Node / Python / Rust without CGO. Recommendation: spawn
  `lyra-server` as a child process and connect via Unix socket. Saves
  language interop pain at the cost of one extra process.
- **Hot-swap transports at runtime?** E.g. embedded client falls back
  from in-process to HTTP if the user opens a "connect to remote
  workspace" dialog. Possible — `Transport` is just an interface — but
  no UI for it yet.

## 11. Cheat sheet

```
Same Go binary, both ends Go         →  InProcessTransport (no codegen, ~50ns)
Wails / Tauri / Electron + Go core   →  WailsTransport       (no Bearer, ~30µs)
Cross-process, same machine          →  SocketTransport      (no Bearer, ~30µs)
Different machine, any language      →  FetchTransport (HTTP)(Bearer, ~200µs+)
Browser, any backend                 →  FetchTransport (HTTP)(only option)
```

One `Transport` interface, six implementations, nine deployment cells.
Pick the cheapest box that crosses your boundary.
